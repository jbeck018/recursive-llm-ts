# Token Tracking and Context Efficiency Guide

## Overview

RLM natively tracks token usage across all completion paths and provides 6 context reduction strategies that provably use fewer tokens than passing large documents as raw context. This guide covers:

1. [Native Token Tracking](#native-token-tracking) -- How token usage flows through the system
2. [Running the Token Tests](#running-the-token-tests) -- How to run and interpret the test suite
3. [Strategy Comparison](#strategy-comparison) -- Benchmark results for all 6 strategies
4. [Recommendations by Document Size](#recommendations-by-document-size) -- Which strategy to use when

## Native Token Tracking

### What Gets Tracked

Every LLM API call made by the Go engine captures the `usage` field from the OpenAI-compatible response:

| Field | Description |
|-------|-------------|
| `total_tokens` | Sum of prompt + completion tokens |
| `prompt_tokens` | Tokens sent to the model (context + query + system prompt) |
| `completion_tokens` | Tokens generated by the model |

Token counts accumulate across **all** LLM calls within a single completion request, including:
- Regular completion iterations (RLM loop)
- Meta-agent query optimization calls
- Context overflow reduction calls (mapreduce, chunked, refine strategies)
- Structured output validation retries
- Parallel subtask execution in structured completions

### Accessing Token Usage in TypeScript

Token fields are returned in `RLMStats` from every completion method:

```typescript
import { RLM } from 'recursive-llm-ts';

const rlm = new RLM('gpt-4o-mini', { api_key: process.env.OPENAI_API_KEY });

const result = await rlm.completion('Summarize this report', largeDocument);

// Direct access
console.log(result.stats.total_tokens);      // e.g., 2450
console.log(result.stats.prompt_tokens);      // e.g., 2100
console.log(result.stats.completion_tokens);  // e.g., 350

// Formatted output
const formatted = rlm.formatResult(result);
console.log(formatted.prettyStats());
// "LLM Calls: 2 | Iterations: 2 | Depth: 0 | Tokens: 2450 (prompt: 2100, completion: 350)"

console.log(formatted.toMarkdown());
// | Metric | Value |
// |--------|-------|
// | LLM Calls | 2 |
// | Iterations | 2 |
// | Depth | 0 |
// | Total Tokens | 2450 |
// | Prompt Tokens | 2100 |
// | Completion Tokens | 350 |
```

Structured completions track tokens identically:

```typescript
const result = await rlm.structuredCompletion('Extract key metrics', report, schema);
console.log(result.stats.total_tokens); // Includes all LLM calls: validation, retries, parallel subtasks
```

### Accessing Token Usage in Go

When using the Go package directly:

```go
engine := rlm.New("gpt-4o-mini", rlm.Config{APIKey: "..."})
result, stats, err := engine.Completion(query, context)

fmt.Printf("Total tokens: %d\n", stats.TotalTokens)
fmt.Printf("Prompt tokens: %d\n", stats.PromptTokens)
fmt.Printf("Completion tokens: %d\n", stats.CompletionTokens)
```

### Backward Compatibility

Token fields use `omitempty` in JSON serialization. When an API provider does not return usage data (rare, but possible with some self-hosted endpoints), these fields are simply absent from the response rather than present as zeros. Existing consumers that don't reference token fields see no change.

## Strategy Comparison

### The 6 Context Reduction Strategies

RLM provides 6 strategies for handling documents that exceed the model's context window. They fall into two categories:

**Algorithmic (no LLM calls, zero cost, fast):**

| Strategy | How It Works | Strengths | Weaknesses |
|----------|-------------|-----------|------------|
| **tfidf** | Splits into sentences, scores each by TF-IDF importance, selects top-k sentences to fill budget | Fast, preserves key terminology, deterministic | May miss context that requires cross-sentence understanding |
| **textrank** | Builds cosine-similarity graph over TF-IDF sentence vectors, runs PageRank, selects top-ranked sentences | Better at finding globally important sentences than TF-IDF | Slower than TF-IDF (O(n^2) graph construction) |
| **truncate** | Keeps first 2/3 and last 1/3 of context, drops middle | Fastest strategy, preserves intro/conclusion, addresses "lost in the middle" | Blindly drops content regardless of importance |

**LLM-based (uses API calls, costs tokens, higher quality):**

| Strategy | How It Works | Strengths | Weaknesses |
|----------|-------------|-----------|------------|
| **mapreduce** (default) | Chunks context, summarizes each chunk in parallel via LLM, combines summaries | Parallel = fast, captures nuance, handles diverse content | Costs tokens for summarization, requires API access |
| **chunked** | Chunks context, extracts relevant info from each chunk in parallel, joins results | Query-focused extraction, parallel | Same cost concerns as mapreduce, may miss globally relevant context |
| **refine** | Processes chunks sequentially, each refining the previous answer | Highest information fidelity, cumulative context | Sequential = slow, highest token cost |

### Benchmark Results

Results from `TestTokenEfficiency_AllStrategiesCompared` with a 40k token document against a 32k model limit:

| Strategy | Tokens Used | Reduction | API Calls | Cost |
|----------|------------|-----------|-----------|------|
| raw (no reduction) | 40,010 | 0% | 0 | $0 |
| **tfidf** | 31,623 | 21.0% | 0 | $0 |
| **textrank** | 31,558 | 21.1% | 0 | $0 |
| **truncate** | 27,244 | 31.9% | 0 | $0 |
| **mapreduce** | varies | 40-80% | N chunks | ~$0.01-0.05 |
| **chunked** | varies | 40-80% | N chunks | ~$0.01-0.05 |
| **refine** | varies | 40-80% | N chunks (sequential) | ~$0.01-0.05 |

### Scaling Behavior

From `TestTokenEfficiency_VeryLargeContext_100kTokens`:

| Input Size | TF-IDF Output | Reduction | TextRank Output | Reduction |
|-----------|--------------|-----------|----------------|-----------|
| ~35k tokens | ~31.6k | 10.2% | ~31.6k | 10.3% |
| ~40k tokens | ~31.6k | 21.0% | ~31.6k | 21.1% |
| ~100k tokens | ~31.6k | 68.4% | ~31.6k | 68.4% |

The algorithmic strategies scale linearly -- they always compress to fit within the token budget regardless of input size. The larger the input, the greater the percentage reduction.

## Recommendations by Document Size

### Small Documents (< 8k tokens)

**Recommendation: No strategy needed (pass raw context)**

```typescript
const rlm = new RLM('gpt-4o-mini'); // 128k limit
const result = await rlm.completion('Summarize', smallDoc);
```

Documents under 8k tokens fit comfortably within any modern model's context window. RLM's pre-emptive overflow check will correctly skip reduction. Token tracking still works -- you'll see the exact token usage in `result.stats`.

### Medium Documents (8k - 32k tokens)

**Recommendation: Use `tfidf` strategy as a safety net**

```typescript
const rlm = new RLM('gpt-4o-mini', {
  context_overflow: {
    enabled: true,
    strategy: 'tfidf',
    safety_margin: 0.15,
  },
});
```

Most modern models (GPT-4o, Claude 3.5, Llama 3.1+) handle 128k+ tokens, so 8-32k documents usually pass through without reduction. But if you're using older models (GPT-3.5-turbo at 16k, GPT-4 at 8k, Llama 3 at 8k), the pre-emptive check will automatically reduce context using the configured strategy.

TF-IDF is recommended here because:
- Zero cost (no API calls)
- Fast (< 100ms for 32k tokens)
- Deterministic (same input = same output)
- Preserves the most informative sentences

### Large Documents (32k - 100k tokens)

**Recommendation: Use `tfidf` for cost-sensitive workloads, `mapreduce` for quality-sensitive workloads**

For cost-sensitive or high-throughput scenarios:

```typescript
const rlm = new RLM('gpt-4o-mini', {
  context_overflow: {
    enabled: true,
    strategy: 'tfidf',  // Zero API cost
    safety_margin: 0.15,
  },
});
```

For quality-sensitive scenarios where spending a few cents per document is acceptable:

```typescript
const rlm = new RLM('gpt-4o-mini', {
  context_overflow: {
    enabled: true,
    strategy: 'mapreduce',  // Parallel LLM summarization
    safety_margin: 0.15,
  },
});
```

At this size range, even with 128k-limit models, the pre-emptive check might trigger if your query + system prompt + response budget push the total over the limit. The key tradeoff:

| Factor | tfidf / textrank | mapreduce |
|--------|-----------------|-----------|
| Cost | $0 | ~$0.01-0.03 per document |
| Speed | < 200ms | 2-10s (parallel API calls) |
| Quality | Good for factual/data-heavy docs | Better for nuanced/narrative content |
| Determinism | Yes | No (LLM variance) |

### Very Large Documents (100k+ tokens)

**Recommendation: Use `mapreduce` with pre-emptive reduction**

```typescript
const rlm = new RLM('gpt-4o-mini', {
  context_overflow: {
    enabled: true,
    strategy: 'mapreduce',
    safety_margin: 0.15,
    max_reduction_attempts: 3,
  },
});
```

At 100k+ tokens, you're dealing with book-length content. Key considerations:

- **Even 128k-limit models may overflow** once you add system prompts, query, and response budget.
- **TF-IDF/TextRank still work** and achieve 68%+ reduction (proven by tests), but extractive compression may lose important cross-paragraph relationships.
- **MapReduce excels** because each chunk gets a full LLM summarization pass that captures nuance, and summaries are combined. If the combined summaries still overflow, the reducer recursively applies another round.
- **Refine is the highest-fidelity option** for cases where cumulative context matters (e.g., legal documents, research papers with building arguments). But it's sequential and costs the most tokens.

For extreme cases, force a model token limit to ensure reduction always triggers:

```typescript
const rlm = new RLM('gpt-4o-mini', {
  context_overflow: {
    enabled: true,
    strategy: 'mapreduce',
    max_model_tokens: 32768, // Force reduction even on 128k models
    safety_margin: 0.15,
  },
});
```

### Decision Matrix

| Document Size | Budget-Conscious | Quality-Focused | Maximum Fidelity |
|--------------|-----------------|----------------|-----------------|
| < 8k | No strategy needed | No strategy needed | No strategy needed |
| 8k - 32k | `tfidf` | `tfidf` | `tfidf` |
| 32k - 100k | `tfidf` | `mapreduce` | `refine` |
| 100k+ | `textrank` | `mapreduce` | `refine` |

### Monitoring Token Usage

Use token tracking to validate that your strategy choice is actually saving tokens:

```typescript
const result = await rlm.completion('Analyze this report', veryLargeDoc);

console.log(`Total tokens used: ${result.stats.total_tokens}`);
console.log(`LLM calls made: ${result.stats.llm_calls}`);

// Compare: what would raw submission cost?
// Estimation: document chars / 3.5 = approximate prompt tokens
const rawEstimate = Math.ceil(veryLargeDoc.length / 3.5);
const actualTokens = result.stats.total_tokens;
const savings = ((rawEstimate - actualTokens) / rawEstimate * 100).toFixed(1);
console.log(`Estimated savings: ${savings}% fewer tokens than raw submission`);
```

## Architecture Notes

### How Token Tracking Flows Through the System

```
OpenAI-compatible API response
  └─ { "usage": { "prompt_tokens": N, "completion_tokens": M, "total_tokens": N+M } }
       │
       ▼
CallChatCompletion() → ChatCompletionResult { Content, Usage *TokenUsage }
       │
       ▼
RLM.callLLM() accumulates into r.stats.{TotalTokens, PromptTokens, CompletionTokens}
       │
       ├─ Also reported to Observer.LLMCall(model, msgCount, tokensUsed, duration, err)
       │
       ▼
JSON IPC stdout → { "stats": { "total_tokens": N, "prompt_tokens": M, "completion_tokens": K } }
       │
       ▼
TypeScript RLMStats interface → result.stats.total_tokens
```

### Token Estimation Heuristic

For pre-emptive overflow detection, RLM uses a character-based estimation since it doesn't bundle a tokenizer:

```
estimated_tokens = ceil(len(text) / 3.5)
```

This is intentionally conservative (over-estimates slightly) to avoid overflow. The ratio of ~3.5 chars/token is based on the `cl100k_base` encoding used by GPT-4/GPT-4o models, averaged across English text and JSON/code content.

### Model Token Limits

Known model limits are built into the engine for pre-emptive overflow detection:

| Model Family | Context Window |
|-------------|---------------|
| GPT-4o, GPT-4o-mini | 128,000 |
| GPT-4-turbo | 128,000 |
| GPT-4 | 8,192 |
| GPT-3.5-turbo | 16,385 |
| Claude 3/3.5/4 | 200,000 |
| o1, o3-mini | 200,000 |
| Llama 3.1+ | 128,000 |
| Mistral Large/Small | 128,000 |

Override with `max_model_tokens` for custom/self-hosted models:

```typescript
const rlm = new RLM('my-custom-model', {
  context_overflow: {
    enabled: true,
    max_model_tokens: 16384, // Your model's actual limit
  },
});
```
